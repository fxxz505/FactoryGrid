# 模块化自动工厂模拟器项目规格说明

## 1. 项目定位

项目暂定名：**FactoryGrid**

FactoryGrid 是一个基于浏览器的模块化自动工厂搭建与仿真平台。用户在无限画布上放置机器、传送带、管道、电网、仓储与逻辑控制器，设计从原料采集到产品交付的完整自动化工厂。系统需要具备类似数字电路仿真器的“元件库 + 连线规则 + 实时仿真 + 调试面板 + 自定义封装 + 示例工程”结构，但主题从数字信号扩展为物流、能量、流体、配方和控制信号。

核心目标不是做一个简单的放置游戏，而是做一个可持续扩展的“工厂系统编辑器”。它应支持用户从零建立自己的生产体系：自定义资源、机器、配方、模块、蓝图和关卡目标。

## 2. 成熟产品参考

### 2.1 Factorio

借鉴点：

- 生产链从采矿、冶炼、组装、科研逐层扩展。
- 传送带、机械臂、箱子、机器、电力构成基础自动化语言。
- 蓝图系统支持复制、复用和规模化建厂。
- 电路网络允许玩家用信号控制设备行为。
- 科技树把“做出产品”转化为“解锁新系统”的长期目标。

适配到本项目：

- 建立资源、配方、机器、物流、控制、科技六大核心系统。
- 把蓝图作为一等功能，而不是后期附加功能。
- 用简化版信号网络实现高自由度自动化控制。

### 2.2 Shapez / Shapez 2

借鉴点：

- 把工厂复杂度压缩到“输入、切割、旋转、堆叠、染色、输出”的清晰规则。
- 强调无限画布、网格放置、传送带流动和蓝图重构。
- 用户目标明确：生产指定形状，提升吞吐。

适配到本项目：

- 初版不追求过多真实工业细节，而是优先确保规则清晰、反馈即时。
- 通过订单目标驱动玩家持续优化产线。
- 分析面板重点显示吞吐、瓶颈和停机原因。

### 2.3 Satisfactory

借鉴点：

- 电力、矿点、机器效率、物流距离共同塑造工厂布局。
- 设备有输入、输出、功率、生产速率和配方配置。
- 里程碑式解锁推动玩家从手工到自动化再到大型工厂。

适配到本项目：

- 每个机器都应有功耗、速率、缓存、输入输出端口。
- 供电不足、输入不足、输出堵塞都要成为可视化状态。
- 用项目里程碑替代单纯分数。

### 2.4 Mindustry

借鉴点：

- 资源采集、传送带物流、生产、防御和核心基地构成闭环。
- 关卡目标和敌人压力让工厂系统有外部消耗。
- 逻辑处理器和单位控制提供高阶自动化。

适配到本项目：

- 后续可加入“任务压力”：订单倒计时、灾害、设备损坏、需求波动。
- 初版可先做“交付中心”作为资源消耗点，形成生产闭环。

## 3. 产品愿景

用户打开应用后，第一屏就是可操作的工厂编辑器：

- 左侧是机器/物流/能源/控制元件库。
- 中间是无限网格画布。
- 右侧是选中对象属性、配方配置和状态调试。
- 底部是仿真时间轴、吞吐统计、错误列表和事件日志。

用户可以从一个矿点开始，逐步搭建：

1. 采矿机采集铁矿。
2. 传送带运输到熔炉。
3. 熔炉消耗电力，把铁矿变成铁板。
4. 组装机消耗铁板和齿轮，生产零件。
5. 交付中心接收零件，产生研究点。
6. 科技树解锁更高级机器。
7. 用户把一段产线封装为蓝图，复用到更大工厂。
8. 分析面板指出瓶颈，用户继续优化吞吐。

这就是项目的核心功能闭环。

## 4. 目标用户

- 喜欢 Factorio、Shapez、Satisfactory、Mindustry 的自动化玩家。
- 需要教学演示的工程、物流、系统设计课程。
- 想学习数据流、状态机、调度、仿真和可视化系统的开发者。
- 喜欢搭建复杂系统并逐步优化的创作者。

## 5. 技术栈

### 5.1 前端

- **Vue 3 + TypeScript**：组件化 UI、状态响应式、易于拆分编辑器面板。
- **Vite**：快速开发与构建。
- **Pinia**：管理工厂工程、仿真状态、用户设置、蓝图库。
- **Canvas 2D**：渲染主画布、网格、机器、传送带、流动物品和选择框。
- **Web Worker**：运行仿真 tick，避免复杂工厂卡住 UI。
- **VueUse**：快捷处理快捷键、窗口尺寸、本地存储等交互细节。

### 5.2 测试

- **Vitest**：测试配方计算、物流传递、仿真 tick、序列化。
- **Playwright**：测试编辑器核心流程，例如放置机器、连接传送带、启动仿真、保存蓝图。

### 5.3 数据存储

初版使用纯前端本地存储：

- `localStorage`：保存用户设置和最近工程。
- `IndexedDB`：保存大型工程、蓝图库、示例工程。
- JSON 文件导入/导出：便于分享和版本管理。

后续可选后端：

- Node.js + Fastify：用户账号、云端工程、蓝图市场。
- PostgreSQL：工程元数据、公开蓝图、版本历史。
- 对初版不是必要项。

## 8. 核心数据模型

### 8.1 Resource 资源

表示可被生产、运输、消耗的物品或介质。

字段建议：

- `id`：唯一标识，例如 `iron_ore`。
- `name`：显示名。
- `category`：`solid | fluid | energy | signal`。
- `stackSize`：单格缓存上限。
- `color`：传送带和统计图颜色。
- `icon`：图标资源。

### 8.2 Recipe 配方

定义输入资源如何转化为输出资源。

字段建议：

- `id`
- `name`
- `durationTicks`
- `inputs`
- `outputs`
- `powerCost`
- `allowedMachineTypes`
- `byproducts`
- `unlockedBy`

### 8.3 Entity 实体

画布上的所有对象都统一为实体。

字段建议：

- `id`
- `type`
- `definitionId`
- `position`
- `rotation`
- `ports`
- `state`
- `config`
- `runtime`

实体类型：

- `resource_node`：矿点、油井、水源。
- `machine`：采矿机、熔炉、组装机、化工机。
- `transport`：传送带、分拣器、管道、分流器。
- `storage`：箱子、仓库、罐体。
- `power`：发电机、电线杆、电池、变压器。
- `control`：传感器、比较器、逻辑门、信号线。
- `sink`：交付中心、研究中心、订单终端。
- `decorative`：标签、区域框、说明文本。

### 8.4 Port 端口

实体之间不直接“随便连接”，而是通过端口连接。

字段建议：

- `id`
- `kind`：`item | fluid | power | signal`
- `direction`：`input | output | bidirectional`
- `localPosition`
- `acceptedResources`
- `rateLimit`

### 8.5 Connection 连接

表示传送带、管道、电缆、信号线等连接关系。

字段建议：

- `id`
- `kind`
- `fromPort`
- `toPort`
- `path`
- `capacity`
- `runtime`

### 8.6 Blueprint 蓝图

蓝图是被选中实体和连接的可复用封装。

字段建议：

- `id`
- `name`
- `description`
- `entities`
- `connections`
- `bounds`
- `requiredResearch`
- `tags`
- `version`

## 9. 模块设计与实现思路

### 9.1 编辑器画布模块

职责：

- 无限网格画布。
- 平移、缩放、框选、多选。
- 放置、移动、旋转、删除实体。
- 端口高亮、拖拽连接、路径预览。
- 撤销/重做。

实现思路：

- 使用 Canvas 2D 负责高频绘制。
- Vue 只负责面板和工具状态，不直接渲染大量实体 DOM。
- 维护 `camera`，包含 `x`、`y`、`zoom`。
- 所有实体坐标使用世界坐标，鼠标交互时做屏幕坐标与世界坐标转换。
- 实体放置采用网格吸附，传送带和管道路径采用正交折线。
- 编辑操作走 Command 模式，支持撤销/重做。

闭环：

用户从元件库拖出机器到画布，画布创建实体，实体进入项目图结构，仿真引擎在下一 tick 读取实体并计算状态，渲染器再把状态反馈到画布。

### 9.2 元件库模块

职责：

- 按分类展示机器、物流、能源、控制、存储、目标设施。
- 支持搜索、筛选、收藏。
- 显示元件说明、输入输出端口、功耗、速率、可用配方。

实现思路：

- 元件定义放在 `src/data/machines.ts`。
- 每个元件定义只描述静态能力，不保存运行状态。
- 元件库读取科技树状态，只显示已解锁项，未解锁项可灰显。

闭环：

科技树解锁新元件，元件库显示新元件，用户放置后参与生产链，生产链交付目标资源又产生新的科技点。

### 9.3 资源与配方模块

职责：

- 定义所有资源和配方。
- 校验机器是否支持某个配方。
- 计算输入是否满足、输出是否堵塞、生产进度。

实现思路：

- 配方系统保持纯函数化，方便测试。
- 每台生产机器运行时保存 `activeRecipeId`、`progressTicks`、`inputBuffer`、`outputBuffer`。
- tick 时先检查输入和电力，再推进进度。
- 进度完成后原子地扣除输入并写入输出。

闭环：

物流系统把资源送入机器输入缓存，配方系统将其转为输出，物流系统继续送往下一台机器或交付中心。

### 9.4 传送带物流模块

职责：

- 处理固体物品在传送带、分流器、合流器、机械臂、仓库之间移动。
- 显示物品流动动画。
- 判断堵塞、断线、方向冲突。

实现思路：

- 初版采用离散格模型：每段传送带按格位保存 item。
- 每个 tick 从下游到上游更新，减少同一 tick 内重复移动。
- 传送带有方向、速度、容量。
- 分流器根据策略把输入分配到多个输出：轮询、优先、过滤。
- 机械臂用于在相邻实体之间转移物品。

闭环：

资源节点产生原料，传送带把原料送入机器，机器产出物品，传送带继续运输，若下游堵塞则上游逐级停机，分析模块标记瓶颈。

### 9.5 管道与流体模块

职责：

- 处理液体和气体的流量、容量、压力简化模型。
- 支持泵、阀门、储罐、流体机器。

实现思路：

- 初版不做复杂流体力学，采用网络均衡模型。
- 将连通管道构成一个 fluid network。
- 每个网络保存总容量、当前流体类型、总量。
- 泵提供方向性流量，阀门控制开关，机器按 tick 请求或注入流体。

闭环：

油井或水源向管网注入流体，泵和管道输送到化工机，化工机产出新资源，储罐缓冲波动，控制信号可开关阀门。

### 9.6 电力模块

职责：

- 发电、输电、储能、耗电。
- 判断供电范围、功率不足、停机。

实现思路：

- 电线杆、电缆连接形成 power network。
- 发电机提供 `generation`，机器报告 `demand`。
- 如果供电不足，按优先级或比例降低机器效率。
- 电池在发电过剩时充电，供电不足时放电。
- 画布上用颜色显示电网负载：绿色正常、黄色紧张、红色不足。

闭环：

燃料或太阳能产生电力，电网供给机器，机器生产资源，资源又可作为发电燃料或解锁更高级电力设备。

### 9.7 控制信号模块

职责：

- 提供高自由度自动化控制。
- 支持传感器、比较器、逻辑门、计数器、信号线。

实现思路：

- 参考电路仿真思路，但信号值不只限于 0/1，可支持数字和资源计数。
- 信号线连接控制端口。
- 传感器读取箱子库存、传送带物品、机器状态、电网负载。
- 比较器输出布尔信号。
- 逻辑门组合信号。
- 执行器控制机械臂、阀门、机器启停、分流器策略。

闭环：

传感器读取工厂状态，逻辑网络计算控制信号，执行器改变物流和机器行为，新的行为改变库存和吞吐，传感器再次读取，形成自动控制闭环。

### 9.8 仿真引擎模块

职责：

- 驱动所有系统按 tick 运行。
- 支持开始、暂停、单步、倍速。
- 确保模块之间更新顺序稳定。

建议 tick 顺序：

1. 读取控制信号输入。
2. 更新电力网络。
3. 更新资源节点产出。
4. 更新机器生产进度。
5. 更新传送带物流。
6. 更新管道流体。
7. 更新仓储与交付中心。
8. 运行错误检测。
9. 生成统计数据和事件日志。
10. 输出渲染快照。

实现思路：

- 初版可在主线程运行，工厂规模扩大后迁移到 Web Worker。
- 引擎输入为项目状态快照，输出为 runtime patch。
- UI 不直接修改 runtime，只提交编辑命令。
- 每个系统尽量保持可独立测试。

闭环：

编辑器修改静态结构，仿真引擎计算动态状态，分析和渲染展示结果，用户根据结果继续修改结构。

### 9.9 错误检测模块

职责：

- 实时发现工厂不可运行或低效原因。

错误类型：

- 输入缺失。
- 输出堵塞。
- 电力不足。
- 配方未设置。
- 端口类型不匹配。
- 传送带方向冲突。
- 管道混入不同流体。
- 控制信号循环震荡。
- 机器无可达交付路径。

实现思路：

- 分为结构错误和运行错误。
- 结构错误在编辑时即时检测。
- 运行错误在 tick 后检测。
- 每条错误记录包含实体 id、严重级别、原因、建议修复动作。

闭环：

错误面板点击问题后定位画布对象，用户修复结构或配置，仿真恢复正常，错误从面板消失。

### 9.10 分析与优化模块

职责：

- 帮助用户理解工厂为什么慢、哪里堵、哪里浪费。

指标：

- 每分钟产量。
- 每分钟消耗量。
- 机器利用率。
- 传送带占用率。
- 电网负载率。
- 库存变化曲线。
- 订单完成速度。
- 关键瓶颈实体。

实现思路：

- 仿真引擎每 N tick 采样一次。
- `throughputAnalyzer` 根据端口流量计算产线吞吐。
- `bottleneckAnalyzer` 根据停机原因排序瓶颈。
- 图表可以先用轻量 SVG/Canvas 自绘，后续再引入图表库。

闭环：

用户完成基础生产后，分析模块指出瓶颈，用户扩建或重构产线，吞吐提升，交付目标加速完成。

### 9.11 蓝图与封装模块

职责：

- 将选中区域保存为可复用蓝图。
- 支持预览、旋转、镜像、批量放置。
- 支持导入、导出和示例蓝图库。

实现思路：

- 选区内实体和连接序列化为相对坐标。
- 粘贴时生成新 id，并重新映射连接端口。
- 蓝图保存依赖版本号，便于后续迁移。
- 初版蓝图只在本地保存，后续可做蓝图市场。

闭环：

用户调试出稳定产线后封装为蓝图，在新区域快速复用，工厂规模扩大，新的规模问题再由分析模块反馈。

### 9.12 科技树与目标模块

职责：

- 为沙盒提供成长路径。
- 把生产结果转化为新能力解锁。

实现思路：

- 交付中心定义订单：需要资源、数量、时限、奖励。
- 研究中心消耗科研包或指定产品产生科技点。
- 科技节点解锁新机器、新配方、新物流速度、新控制器。
- 初版提供“自由模式”和“任务模式”两种入口。

闭环：

订单要求推动用户生产指定资源，交付获得研究点，研究解锁新机器，新机器支持更复杂配方，产生更高级订单。

### 9.13 存档、导入导出模块

职责：

- 保存工程。
- 支持分享工程和蓝图。
- 支持自动保存和版本兼容。

实现思路：

- 工程保存为 JSON。
- `project.version` 用于迁移。
- 自动保存间隔 5 到 10 秒。
- 导入时先做 schema 校验，再写入 store。
- 大工程保存到 IndexedDB。

闭环：

用户搭建工厂后自动保存，刷新页面恢复状态；用户导出工程给他人，他人导入后继续仿真和修改。

### 9.14 示例工程与教程模块

职责：

- 降低学习门槛。
- 展示系统能力。

示例工程：

- 铁板基础生产线。
- 齿轮自动化生产线。
- 双输入组装机。
- 电力不足案例。
- 传送带堵塞案例。
- 分流器过滤案例。
- 库存控制信号案例。
- 蓝图复制扩建案例。

实现思路：

- 示例工程就是普通工程 JSON。
- 教程步骤通过高亮实体、锁定工具、提示目标实现。
- 教程不是弹窗说明，而是让用户直接完成操作。

闭环：

用户从示例理解系统，复制修改示例形成自己的工程，再沉淀为自定义蓝图。

## 10. 核心功能闭环

### 10.1 生产闭环

```text
资源节点 -> 采集机器 -> 物流系统 -> 加工机器 -> 成品 -> 交付中心 -> 奖励/研究点
```

这是最基础的游戏与仿真闭环。任何新增模块都应服务于这个闭环。

### 10.2 优化闭环

```text
运行工厂 -> 收集指标 -> 发现瓶颈 -> 修改布局/配方/电力/控制 -> 再次运行
```

这是让项目耐玩的关键。没有分析和反馈，工厂只是静态拼图。

### 10.3 成长闭环

```text
完成订单 -> 获得科技点 -> 解锁新机器/配方 -> 生产更复杂物品 -> 完成更高级订单
```

这是长期目标来源。

### 10.4 复用闭环

```text
搭建产线 -> 调试稳定 -> 保存蓝图 -> 批量复用 -> 扩大规模 -> 产生新优化问题
```

这是让用户从“手动搭建”进入“系统设计”的关键。

### 10.5 控制闭环

```text
传感器读取状态 -> 逻辑网络判断 -> 执行器改变行为 -> 工厂状态变化 -> 传感器再次读取
```

这是高自由度的核心。它让项目从普通工厂模拟变成可编程自动化沙盒。

## 14. 代码风格建议

核心引擎使用纯函数优先，避免 UI 和仿真逻辑混在一起。

示例：

```ts
export interface TickContext {
  tick: number
  delta: number
  project: ProjectState
  runtime: RuntimeState
}

export interface SystemResult {
  runtimePatch: RuntimePatch
  events: SimulationEvent[]
  errors: FactoryError[]
}

export function runRecipeSystem(context: TickContext): SystemResult {
  const patch: RuntimePatch = {}
  const events: SimulationEvent[] = []
  const errors: FactoryError[] = []

  for (const machine of context.project.entities) {
    if (machine.kind !== 'machine') continue
    // 读取机器配置和 runtime，计算生产进度，写入 patch。
  }

  return { runtimePatch: patch, events, errors }
}
```

约定：

- `models/` 只放类型和数据结构。
- `engine/` 不引用 Vue。
- `stores/` 负责连接 UI 和引擎。
- `render/` 只负责绘制，不修改业务状态。
- `data/` 中的元件、资源、配方定义必须可序列化。

## 15. 边界规则

Always：

- 每个核心系统都要有单元测试。
- 仿真逻辑必须和 Vue 组件解耦。
- 工程文件必须可导入导出。
- 所有实体都通过统一 id 引用。
- 每个机器停机都必须能给出原因。

Ask first：

- 引入大型渲染库或游戏引擎。
- 加后端服务。
- 加多人协作。
- 加商业化蓝图市场。
- 改变核心坐标/实体模型。

Never：

- 不把仿真状态散落在 Vue 组件内部。
- 不让传送带、机器、仓库各自私有处理资源流动。
- 不在没有迁移机制的情况下改工程 JSON 格式。
- 不把第一阶段做成过度复杂的真实工业仿真。

## 16. 成功标准

项目第一版完成时应满足：

- 用户能在 5 分钟内完成第一条自动化产线。
- 至少支持 10 种资源、8 种机器、5 种物流元件。
- 1000 个实体以内保持流畅编辑。
- 基础仿真 tick 稳定，没有资源凭空丢失或复制。
- 每个停机状态都有可解释原因。
- 工程可保存、刷新恢复、导入导出。
- 至少 5 个示例工程可一键加载。
- 至少 80% 核心引擎逻辑有单元测试覆盖。

## 19. 推荐第一条用户体验路径

1. 打开应用，自动加载“新手工厂”空白地图。
2. 左侧选择采矿机，放在铁矿点旁边。
3. 放置燃煤发电机和电线杆，采矿机亮起。
4. 从采矿机输出口拖出传送带。
5. 放置熔炉，设置铁板配方。
6. 传送带把铁矿送入熔炉。
7. 熔炉输出铁板。
8. 传送带把铁板送入交付中心。
9. 交付中心进度条增长并完成第一个订单。
10. 系统解锁组装机和齿轮配方。

这条路径能验证编辑、物流、电力、配方、仿真、目标和解锁的最小闭环。
